Table elements

Sometimes a table is the right answer — when readers need to look up exact values. Make it scannable.

Example

North Carolina public schools and student population by urbanicity, 2023–24
Urbanicity Schools Student pop. % of NC student pop.
City 612 472,300 31.4%
Suburb 498 418,900 27.8%
Town 331 186,200 12.4%
Rural 1,074 427,500 28.4%
Total 2,515 1,504,900 100.0%

Source: NCDPI school directory, 2023–24. Counts exclude charters with statewide enrollment.

Anatomy of an NCDPI table

Same idea as the chart anatomy — name each part, then its spec. A table simply has fewer parts. Match these and it reads as a clean look-up grid.

Part Specification
Title (the <caption>) 14 pt, bold, Black #000000 — sized to sit above the header row in the hierarchy
Header row White text on DPI Blue #003A70 — short, strong labels
Row header Leftmost cell of each row; left-aligned, regular weight
Row banding 10% DPI Blue #E6EDF6 on alternate rows — guides the eye across a row without shouting
Dividers Light grey #DEDEDE
Total row 2 px top rule, bold, no fill — separates the summary from the data
Source note 10 pt, Dark grey #525A60 — a table's source note sits one step below the brand's 11 pt chart note, so it recedes

Alignment

  • Numbers align right and share decimal places, so magnitudes line up visually.
  • Text aligns left.
  • Headers match the alignment of the column beneath them.

Make tables do more

  • Sort by the column that matters, not alphabetically by default.
  • Add an embedded sparkline column to show a trend per row without a separate chart.
  • Lightly shade or mark the values that carry the message — don’t make readers hunt.

A few small in-cell techniques — one row each. They borrow the chart palette so the table still reads as NCDPI:

In-cell techniques (illustrative)
Technique Example Reach for it when…
Sparkline each row has a short trend (say, attendance 2019–2024) and a full line chart would be overkill. Color the line navy #003A70; dot the last point.
In-cell data bar 86   readers compare magnitudes down a column at a glance — the bar length does work the bare number can't. Keep the value visible alongside it.
Heat cell 12 47 93 spotting highs and lows across a grid. Shade with Sequence to Neutral (light grey → #5085BC), dark = higher. Mind contrast if you also print the value: dark ink on the darkest steps fails WCAG AA for small text (~4.2:1), and white text is worse — so keep the in-cell number only where the shade is light, or show the value outside the cell.

One enrichment per table, not all three at once. These are accents on a look-up grid — if every column has a bar or a heat tint, you’ve built a cluttered chart, not a table.

Accessibility

Most NCDPI tables are built in Tableau, PowerPoint, or Word — not hand-coded. The principles here apply in every tool: give the table a real title/caption, use genuine header rows (not bold text faking a header), keep one value per cell (avoid merged cells), and never use a table just for page layout. The HTML attributes shown below (scope, headers/id) are how you satisfy these when you hand-code a table — in Tableau and Office, use the built-in table, crosstab, and caption features and they handle most of it for you.

A data table is one of the few places where structure directly affects whether a screen-reader user can use the table at all — so the underlying markup (or your tool’s accessibility features) matters more here than it does for a chart.

  • Use <th scope="col"> on column headers (and scope="row" on the leading cell of each row where it makes sense). WCAG 2.1 SC 1.3.1 Info and Relationships requires that the table’s structure be programmatically conveyable — scope is how you do that.
  • Give every table a <caption> or aria-label. The caption answers “what is this table about?” before the screen reader starts reading cells.
  • Avoid merged header cells. WCAG doesn’t ban them outright, but screen-reader support for merged cells is uneven, and they make linear navigation confusing. If you truly need a merged header (a complex multi-level table), mark it up with explicit headers / id attributes so each data cell points to the headers that govern it.
  • Don’t use a table for layout. If the goal is page layout, use CSS — not <table>.

A simple, well-marked-up table beats a clever, complex one every time. If you find yourself reaching for merged cells, ask whether two simple tables would tell the same story better.

Pair charts with an accessible table

Even a perfect chart leaves screen-reader users out of the conversation. For any complex chart in a dashboard or slide deck, give them a way to reach the underlying numbers.

  • In Tableau: add a “View data” button that toggles a hidden cross-tab via dynamic zone visibility. Default-hidden so it doesn’t crowd the visual; one click for users who need it.
  • In PowerPoint: put the chart on one slide and add a “Data table” hyperlink to a companion slide that holds the corresponding cross-tab.
  • In a web report: offer a “Show table” toggle beneath the chart, or link to a downloadable CSV.

A table answers “what is the exact value?”; a chart answers “what is the pattern?” If your reader needs both — or if accessibility requires both — give them both.
